# Amazon Simple Storage Service

## S3 Introduction

- **What is Amazon S3?**
  - Object storage service
  - Scalability, availability, security, performance
  - Store and protect data for many use cases
- Core AWS building block
  - Often described as effectively unlimited scale for typical workloads
- **How it works**
  - Data stored as **objects** inside **buckets**
  - Object = file + metadata
  - Each object has a **key** (identifier within the bucket)
- **Consistency**
  - Strong **read-after-write** consistency for `PUT` and `DELETE` on objects
  - Reads of ACLs, tags, metadata, S3 Select: strongly consistent
- Many AWS services integrate with S3
  - Analytics
  - ML
  - Logging [1]

### S3 Use Cases

Examples from AWS documentation:

- Data lakes
- Big data analytics
- Websites and static content
- Mobile applications
- Backup and restore
- Archive
- Enterprise applications
- IoT devices

Related product themes:

- Management features to optimize storage
- Organize and configure access
- Meet business and compliance requirements [1]

## Buckets

- **Container** for objects
  - Region-scoped
  - At creation: choose **bucket name** and **AWS Region**
  - After creation: **name** and **Region** cannot be changed
- **Types**
  - Each type targets different patterns
  - General purpose
  - Directory
  - Table
  - Vector
- **General purpose** (default for most cases)
  - Objects across storage classes (except S3 Express One Zone)
  - Redundancy across AZs
  - **Global namespace** per partition: bucket name unique across all accounts in that partition
- Optional **account regional namespace**
  - Names reserved for your account only
- **Directory buckets**
  - Low latency
  - Data residency
  - Hierarchical prefixes
  - S3 Express One Zone in AZ directory buckets
- **Table buckets**
  - Tabular data
  - Apache Iceberg
  - Analytics and ML
- **Vector buckets**
  - Vector embeddings
  - Similarity search
- Buckets **organize** the S3 namespace
- Buckets **identify** the account billed for storage and data transfer
- Buckets **aggregate** usage reporting [1]

- **Private by default**
  - Access only when you grant it
  - Policies
  - IAM
  - ACLs
  - Access points [3]

- **Block Public Access**
  - Four settings
  - On by default for new general purpose buckets
  - Can override policies and ACLs to cap public exposure [2]

## Objects

- **Fundamental** unit stored in S3
  - **Data**
  - **Metadata**: name-value pairs
    - System default
    - HTTP-style fields (e.g. `Content-Type`)
    - Optional user-defined metadata
- **Identity** in a bucket
  - **Key**
  - **Version ID** (when Versioning is on)
- **Key**: name used to retrieve the object
  - **Prefix** + **object name**
  - `s3://my-bucket/wielki/mistrz/ulryk.txt`
  - **Prefix**: `wielki/mistrz/`
  - **Object name**: `ulryk.txt`
- **Value**: object bytes
  - Size from zero up to **50 TB**
  - With a single PUT operation, you can upload a single object **up to 5 GB in size**
  - Bigger files can get uploaded using multipart upload operation
- **Subresources** (e.g. ACL, legal hold): attach to bucket or object; not standalone
- **Access control**
  - Resource-based: ACL, bucket policy
  - Identity-based: IAM
  - **Private by default**
  - Presigned URLs for time-limited sharing
- **Tags**: optional categorization
  - Access control
  - Cost allocation [4]

## Static Website Hosting

- S3 can host static websites and have them accessible on the Internet
- The website URL will be (depending on the region)
  - http://bucket-name.s3-website-aws-region.amazonaws.com 
  - http://bucket-name.s3-website.aws-region.amazonaws.com
- If you get a 403 Forbidden error, make sure the bucket policy allows public reads!

## Versioning

- **What it is**
  - Keeps **multiple variants** of an object under the **same key** in one bucket
  - Preserve, retrieve, and restore **every version** of stored objects
  - Helps recover from unintended overwrites and application failures
- **Scope**
  - Turned **on or off at the bucket level** (versioning state applies to **all** objects in that bucket)
  - **Disabled by default**; you must **explicitly enable** versioning
- **States**
  - **Unversioned** (default)
  - **Versioning-enabled**
  - **Versioning-suspended**
  - After a bucket is version-enabled, it **cannot** return to **unversioned**; you may only **suspend** versioning
- **Version IDs**
  - New and overwritten objects get a **unique version ID**
  - Concurrent writes to the same key: **all** versions are stored
- **Objects already in bucket before versioning**
  - Keep **version ID `null`** until modified; enabling versioning does **not** rewrite existing objects in place
- **Overwrite**
  - New `PUT` for same key adds a **new object version**
- **Delete**
  - Simple `DELETE` without version ID inserts a **delete marker** (current version becomes the marker); older versions can remain recoverable
- **Suspend versioning**
  - **Does not remove** existing object versions; changes how **future** requests are handled
- **Cost**
  - Normal S3 rates for **each stored version** (each version is a full object, not a diff)
- **Best practice**
  - Often recommended for production buckets that need recovery options; pair with **Lifecycle** for noncurrent version costs [5]

## Replication

- **What it is**
  - **Automatic**, **asynchronous** copy of objects between buckets
  - Source and destinations can be **same account** or **different accounts**
  - One destination or **multiple** destination buckets; destinations **same Region** or **different Regions**
- **Types**
  - **Live replication**: new and updated objects after the rule exists (**CRR** or **SRR**)
  - **On-demand**: **S3 Batch Replication** for objects that **already existed** before live rules, failed replicas, or other batch scenarios
- **CRR (Cross-Region Replication)**
  - Different **AWS Regions**
  - Use cases: **compliance** (greater distance between copies), **lower latency** for users in another geography, **operational efficiency** (same data in two Regions for compute)
- **SRR (Same-Region Replication)**
  - Same **AWS Region**
  - Use cases: **log aggregation** into one bucket, **production and test** account replication with metadata preserved, **data sovereignty** (multiple copies in separate accounts within a Region)
- **Requirements**
  - **Versioning enabled** on **both** source and destination buckets
  - **IAM role** (and policies) so S3 can replicate **on your behalf**
  - Cross-account: **destination** account often grants **source** via **bucket policy**
  - Destination cannot be **Requester Pays**; **Object Lock** on source implies Object Lock on destination (and some changes only via API/CLI/SDK)
- **Live replication and existing objects**
  - By default, **does not** backfill objects that existed **before** the replication configuration; use **Batch Replication** for those
- **Delete behavior**
  - `DELETE` **without** version ID: creates **delete marker**; replication of delete markers depends on rule configuration (e.g. **Filter**-based rules vs older **V1**-style behavior) and **cross-account** defaults
  - `DELETE` **with** specific **version ID**: that version is removed in **source**; that deletion is **not** replicated to destination (protects against malicious version deletes)
- **No automatic chaining**
  - Objects in bucket **B** that are **replicas** from bucket **A** are **not** replicated again when **B** is source to **C**; use **Batch Replication** if you need replicas-of-replicas copied onward [6][7][8]
